Process Steps

Domain Input

The process begins by entering a target domain (e.g., example.com). The tool validates the domain format and prepares it for the recon workflow. It's designed to keep things simple ‚Äî just input and go.

02

Subdomain Enumeration

Using passive techniques like querying Certificate Transparency logs (e.g., crt.sh), the tool gathers a list of subdomains associated with the main domain. This helps identify additional services or environments like dev.example.com or api.example.com that might not be publicly advertised.

03
üì°

DNS & WHOIS Lookup

The tool fetches DNS records such as A, MX, TXT, and NS, giving insight into how the domain is structured and where services are hosted. It also performs a WHOIS lookup to gather metadata like domain registration dates, registrar info, and ownership (if public). This information can be useful for identifying shadow IT or abandoned domains.

04
üìÅ

Public File Discovery

Next, the tool checks for commonly exposed public files like /robots.txt, /sitemap.xml, and potentially misconfigured .env or .git folders. These files can accidentally reveal sensitive information, internal paths, or forgotten development setups.

05

Optional Port Scanning (Active)

If the user enables it and has permission, the tool performs a lightweight port scan to identify open services on the target. This includes common ports like 80 (HTTP), 443 (HTTPS), and 22 (SSH). It's useful for building a basic service map but only runs when explicitly allowed to avoid ethical or legal issues.

06
üß†

Technology Fingerprinting

To understand the tech stack, the tool sends HTTP requests and analyzes response headers. This can reveal backend technologies such as Apache, Nginx, PHP, or Node.js ‚Äî which is helpful for identifying potential CVEs or misconfigurations later.

07
üßæ

Output Summary

At the end of the scan, the tool compiles everything into a clean summary. It lists discovered subdomains, open ports (if scanned), DNS records, WHOIS info, and any public files found. This gives the tester a clear starting point for deeper analysis or vulnerability testing.
